# Dataverse

Dataverse is the relational data platform for Power Platform, Dynamics 365, and Copilot Studio solution data.
It combines metadata-driven schema, role-based security, audited operations, API surface area, and ALM integration.

## Dataverse Architectural Fundamentals

- Metadata-first model: tables, columns, relationships, forms, views, and business rules are all metadata artifacts.
- Logical names are immutable identifiers and should be treated like API contracts.
- Display names are mutable and localizable; do not rely on display names in automation.
- Every row has system columns such as `createdon`, `modifiedon`, `ownerid`, `statecode`, and `statuscode` (table dependent).
- All solution-aware artifacts should be managed through managed/unmanaged solutions and source-controlled via unpacked XML.

## Tables (Entities)

### Standard vs Custom Tables

| Category | Description | Example | Governance Impact |
|---|---|---|---|
| Standard table | First-party Microsoft table delivered by Dataverse or Dynamics apps | `account`, `contact`, `incident` | Upgrades may add columns/forms; avoid unsupported customizations to core behavior |
| Custom table | Tenant-defined table for domain-specific data | `cr3d_project`, `cr3d_vendorcontract` | Full ownership of schema and lifecycle; naming discipline is mandatory |

Design guidance:

- Prefer standard tables when they align with domain semantics and process behavior.
- Use custom tables when business concept is unique or standard table baggage is unnecessary.
- Avoid creating near-duplicate custom equivalents for mature standard entities unless isolation is intentional.

### Table Types

| Table Type | Use Case | Storage/Execution Model | Constraints |
|---|---|---|---|
| Standard | Core business transactional data | Full Dataverse relational + security + events | Best default choice for most line-of-business models |
| Activity | Time-bound interactions (tasks, email, phone calls) | Activity pointer model with timeline integration | Activity-specific behaviors and relationships |
| Virtual | Data remains in external source | Read/write proxied through data provider | Limited Dataverse features depending on provider |
| Elastic | High-scale, high-throughput, distributed workloads | Optimized for throughput/partitioning | Feature parity differs from standard tables; validate requirements |

### Naming Conventions

Recommended naming template:

- Publisher prefix: `cr3d` (example)
- Table schema name: `cr3d_Order`
- Table logical name: `cr3d_order`
- Primary name column: `cr3d_name`
- Primary key: `cr3d_orderid`

Naming rules:

1. Use singular nouns for table names (`Project`, not `Projects`).
2. Keep logical names lowercase snake-like style generated by platform and never rename after deployment.
3. Prefix every custom artifact with your publisher prefix.
4. Use stable, API-friendly names (no environment-specific terms like `dev_` or `temp_`).
5. Treat display names as UI only; all automation should use logical names.

### Table Properties

Common table-level properties:

- Ownership: User or Team-owned vs Organization-owned.
- Auditing: Enable at table and column level.
- Change tracking: Required for delta sync patterns.
- Offline availability: Needed for offline client scenarios.
- Duplicate detection: Rules for probabilistic match prevention.
- Notes/activities enablement: Timeline and annotations support.
- Search indexing inclusion: Dataverse Search scope.
- Data retention rules: Lifecycle governance and compliance.

Design decision matrix:

| Decision | Prefer This | When |
|---|---|---|
| Ownership model | User/Team-owned | Need row-level ownership security and assignment workflows |
| Ownership model | Organization-owned | Data is shared globally with no ownership semantics |
| Auditing | Enabled | Regulated processes, approvals, sensitive data changes |
| Change tracking | Enabled | Integration consumers need incremental sync |

## Columns (Fields)

### Column Data Types

| Data Type | Typical Use | Notes |
|---|---|---|
| Single line of text | Names, codes, short labels | Supports formatting variants; use max length governance |
| Multiple lines of text | Descriptions, notes | Consider rich text and search behavior |
| Whole number | Counts, integer states | Use min/max bounds for data quality |
| Decimal number | Precise fractional values | Use precision per domain requirement |
| Currency | Monetary values | Includes exchange rate handling in multi-currency orgs |
| Date and time | Events, due dates, transactions | Choose behavior (user local, date only, timezone independent) carefully |
| Lookup | Referential link to another row | Enforces relationship integrity and navigation |
| Choice | Controlled vocabulary | Use global choice where reuse needed |
| Yes/No | Binary flags | Suitable for explicit booleans only |
| File | Binary documents | Prefer when file is part of record lifecycle |
| Image | Record-associated image | Great for profile/product visuals |

### Column Properties

Important properties to configure explicitly:

- Required level: Business required, recommended, optional.
- Searchable: Whether included in quick find / Dataverse search index config.
- Auditing: Track who changed what and when.
- Security: Field-level security eligible on supported columns.
- Validations: Min/max values, max length, format constraints.
- Default values: Carefully chosen defaults reduce maker logic complexity.
- Behavior for date/time columns: strongest source of downstream defects if misconfigured.

### Calculated Columns

Calculated columns are server-side expressions evaluated on record operations based on other columns.

Use cases:

- SLA target date from created date + priority.
- Derived status labels.
- Numeric computations shared across forms and APIs.

Implementation notes:

- Evaluate server-side consistently across apps.
- Avoid overloading calculated logic with very complex branching when plugin or flow is clearer.

### Rollup Columns

Rollup columns aggregate related table data, often on schedule.

Use cases:

- Total invoice amount per account.
- Count of open cases per customer.
- Last activity date from related activity table.

Caveats:

- Aggregation timing is not always immediate.
- Understand recalculation schedules and on-demand recalc behavior.

### Formula Columns

Formula columns provide expression-based computed values with modern formula authoring.

Use cases:

- Concatenated business labels.
- Conditional categorization logic.
- Lightweight computed values where real-time server evaluation is needed.

## Relationships

### Relationship Types

| Type | Dataverse Semantics | Example |
|---|---|---|
| 1:N | One parent has many children | Account has many Contacts |
| N:1 | Child points to one parent (reverse of 1:N) | Contact belongs to one Account |
| N:N | Many rows on both sides, via intersect relation | User can belong to many Projects and Project has many Users |

### Cascading Behavior

Cascading controls behavior when parent records are assigned, shared, reparented, or deleted.

| Operation | Typical Cascade Options | Design Warning |
|---|---|---|
| Assign | Cascade all, active, none | Unintended ownership reassignment can break security expectations |
| Share/Unshare | Cascade options | Over-sharing risk in sensitive domains |
| Delete | Remove link, restrict, cascade delete | Cascade delete can cause destructive data loss if misconfigured |
| Reparent | Cascade options | Data visibility changes can be broad |

### Relationship Roles

Relationship roles provide semantic role labeling for related connections.

Examples:

- Contact role on account: Decision maker, Finance approver.
- Partner role on opportunity: Reseller, Distributor.

Use roles to improve clarity in relationship-driven business processes and reporting.

## Views

Views define query + projection metadata for model-driven experiences and can influence discovery/search patterns.

### View Types

| View Type | Owner | Purpose |
|---|---|---|
| System view | System customization | Organization-wide canonical filtered lists |
| Personal view | Individual user | User-specific working sets |
| Public view | Shared customization context | Team-shared lists, often curated by admins |
| Quick find view | Table-level search scope | Determines searchable columns in quick find |
| Associated view | Subgrid context | Controls related records shown from parent forms |

### View Design

Best practices:

1. Keep columns task-oriented; avoid overloaded wide views.
2. Index/filter on columns frequently used in where clauses.
3. Use clear display names and predictable sort order.
4. Keep quick find searchable columns bounded to high-value attributes.

Example filter strategy table:

| Scenario | Filter Pattern |
|---|---|
| Active records queue | `statecode = Active` and `ownerid = Current User` |
| SLA breach watch | `duedate < Today` and `status != Closed` |
| Data quality cleanup | `requiredColumn Is Blank` |

## Business Rules

Business rules provide no-code conditional UI/data logic across model-driven forms and server evaluations (scope dependent).

### Conditions

Condition operators commonly include equals, not equals, contains data, does not contain data, greater than, less than.

### Actions

| Action | Behavior | Example |
|---|---|---|
| Set field value | Programmatically assign value | Set `priority` to High when `amount > 100000` |
| Set visibility | Show/hide control | Hide discount section for non-manager role indicators |
| Set business required | Toggle requiredness | Require approval reason when status = Rejected |
| Show error message | Block invalid state | Show error when end date earlier than start date |

### Scope

| Scope Option | Applies To | Guidance |
|---|---|---|
| Entity | Server and all forms where supported | Use for cross-client consistency |
| All forms | All model-driven forms for table | Use when behavior should be UI-wide |
| Specific form | One form | Use for specialized process forms only |

## Choices (Option Sets)

### Local vs Global Choices

| Choice Type | Reusability | Governance |
|---|---|---|
| Local choice | Bound to one table column | Fast to create, can lead to duplication |
| Global choice | Reused across columns/tables | Preferred for enterprise vocabularies |

Recommendations:

- Use global choices for statuses, risk ratings, categories reused in multiple domains.
- Reserve local choices for one-off table semantics.

### Multi-Select Choices

Use multi-select when a record can hold multiple categorical tags.

Caveats:

- Query complexity increases.
- Reporting and integration mappings need explicit handling.

### Dependent Choices

Dependent choices constrain valid child options based on parent selection (for example Country -> State).

Governance note:

- Validate cross-system integrations respect dependency logic to prevent invalid combinations.

## Polymorphic Lookups

Polymorphic lookup columns can target more than one table type.

| Lookup | Targets | Typical Scenario |
|---|---|---|
| Customer | Account or Contact | Case linked to either company or individual |
| Regarding | Multiple activity-enabled tables | Activity references varied business records |
| Owner | User or Team | Ownership/assignment model |

Design considerations:

- API consumers must inspect lookup logical name annotations to determine concrete type.
- UI logic should handle type-specific rendering and commands.

## Virtual Tables

Virtual tables expose external data in Dataverse without physically copying data.

### Integration Models

- OData v4 providers for RESTful external datasets.
- Custom data providers for proprietary/legacy systems.

### When to Use

Use virtual tables when:

- Data residency requires source-of-truth to remain external.
- You need near-real-time access with minimal duplication.
- Basic CRUD and browsing are sufficient for scenario capabilities.

Avoid or validate deeply when:

- Advanced Dataverse features are required (auditing depth, offline, complex transactions, some business logic layers).
- Latency to external source is unpredictable.

## Elastic Tables

Elastic tables are built for very high throughput and large-scale event-like or telemetry-like scenarios.

### Characteristics

- Optimized for scale and write throughput.
- Partition-oriented architecture.
- Intended for patterns where standard relational transaction semantics may not be primary requirement.

### Standard vs Elastic Decision Guide

| Requirement | Standard Table | Elastic Table |
|---|---|---|
| Rich relational behavior | Strong fit | Limited/validate |
| Highest transactional throughput | Moderate | Strong fit |
| Complex model-driven customization | Strong fit | Scenario-dependent |
| Massive event ingestion | Can bottleneck | Strong fit |

## Security

Dataverse security is layered and composable.

### Security Roles

Security roles define privileges (Create, Read, Write, Delete, Append, Append To, Assign, Share) with depth levels.

Depth levels commonly include:

- User (basic)
- Business Unit
- Parent:Child Business Units
- Organization

### Business Units

Business units partition organizational hierarchy and affect role privilege scope.

Design guidance:

- Keep hierarchy aligned to security boundary needs, not org chart volatility.
- Avoid deep hierarchies without clear access rationale.

### Teams

Team patterns:

- Owner teams for shared record ownership.
- Access teams for ad-hoc collaboration without ownership transfer.

### Field-Level Security Profiles

Use field-level security for sensitive columns such as salary, personal identifiers, or regulated values.

Key point:

- Field security protects at column granularity even when table read access exists.

### Row-Level Security (RLS)

Dataverse row access is enforced by ownership + role privilege depth + sharing/team mechanics.

RLS implementation tools:

- Ownership model.
- Role depth.
- Sharing.
- Team ownership.
- Hierarchy security overlays.

### Hierarchy Security

Hierarchy security can grant managers or positional hierarchies visibility into subordinate records.

Use with care:

- Validate for overexposure in matrix organizations.
- Document exception patterns and legal constraints.

## Dataverse Web API

The Dataverse Web API is OData-compliant and supports CRUD, association management, batch, metadata operations, and change tracking.

### CRUD Operations

Create example:

```http
POST https://{org}.crm.dynamics.com/api/data/v9.2/accounts
Content-Type: application/json

{
  "name": "Contoso Manufacturing",
  "telephone1": "+1-555-0100"
}
```

Retrieve example:

```http
GET https://{org}.crm.dynamics.com/api/data/v9.2/accounts({accountid})?$select=name,telephone1,createdon
```

Update example:

```http
PATCH https://{org}.crm.dynamics.com/api/data/v9.2/accounts({accountid})
Content-Type: application/json

{
  "telephone1": "+1-555-9999"
}
```

Delete example:

```http
DELETE https://{org}.crm.dynamics.com/api/data/v9.2/accounts({accountid})
```

### OData Query Patterns

```odata
GET /api/data/v9.2/contacts?$select=fullname,emailaddress1&$filter=statecode eq 0&$orderby=createdon desc&$top=50
```

```odata
GET /api/data/v9.2/accounts?$select=name&$expand=primarycontactid($select=fullname,emailaddress1)
```

```odata
GET /api/data/v9.2/incidents?$select=title,ticketnumber&$filter=contains(title,'urgent')
```

### Batch Requests

Batch allows multiple operations in one HTTP call for throughput and reduced network overhead.

```http
POST /api/data/v9.2/$batch
Content-Type: multipart/mixed;boundary=batch_123

--batch_123
Content-Type: application/http
Content-Transfer-Encoding:binary

GET /api/data/v9.2/accounts?$top=10 HTTP/1.1

--batch_123
Content-Type: application/http
Content-Transfer-Encoding:binary

POST /api/data/v9.2/tasks HTTP/1.1
Content-Type: application/json

{"subject":"Follow up"}
--batch_123--
```

### Change Tracking

Change tracking enables incremental extraction and synchronization patterns.

Pattern:

1. Enable change tracking on table.
2. Query with delta links where supported.
3. Persist token externally.
4. Requery for deltas only.

### Dataverse Web API Performance and Reliability Practices

- Use `$select` aggressively; never pull full rows if only a few columns are needed.
- Use server-side filters before pagination.
- Handle 429 throttling with backoff and retry-after semantics.
- Keep association/disassociation operations explicit and idempotent where possible.
- For integrations, map logical names and option values centrally.

# Power Fx in Copilot Studio

Power Fx is the low-code formula language used in Copilot Studio expressions, conditions, variable assignments, and response shaping.

## Formula Language Fundamentals

- Strongly typed: values carry explicit or inferred types.
- Functional: formulas return values from expressions rather than imperative statements.
- Excel-inspired: function naming and expression composition feel familiar to spreadsheet users.
- Case-sensitive in many practical authoring contexts for identifiers and member access; keep naming exact to avoid runtime errors.
- Immutable expression style: use `With()` and table transformations to avoid opaque side effects.

Copilot Studio-specific principles:

1. Keep formulas short and composable due to authoring constraints.
2. Move reusable logic into variables and scoped records.
3. Validate API response shape before dereferencing nested properties.

## Data Types

| Type | Description | Example |
|---|---|---|
| Text | String content | `"Hello"` |
| Number | Numeric value | `42`, `3.14` |
| Boolean | True/false | `true` |
| DateTime | Date/time value | `Now()` |
| Table | Sequence of records | `Table({Name:"A"},{Name:"B"})` |
| Record | Named fields | `{Name:"A", Score:90}` |
| Choice | Enumerated option value | `Topic.Priority` from choice-backed source |
| Blank | Null-like absence | `Blank()` |
| GUID | Identifier | `GUID("8f0f8f2b-9f1f-4f55-bec6-2cbe2af7f9aa")` |

Type handling guidance:

- Use `Text()`, `Value()`, `Boolean()` conversions when parsing untyped JSON.
- Use `Coalesce()` to provide defaults for blanks.
- Prefer explicit casting in branch outputs to keep type consistency.

## Variable Scopes in Copilot Studio

| Scope | Prefix | Lifetime | Typical Use |
|---|---|---|---|
| Topic | `Topic.VarName` | Current topic/dialog execution | Intermediate calculation and branch state |
| Global | `Global.VarName` | Across topics in session | Shared user context or flow output reuse |
| System | `System.*` | Runtime-provided | User profile, locale, channel metadata |
| Environment | `Environment.*` | Environment configuration | App-level settings, endpoints, feature flags |

Examples:

```powerfx
Topic.GreetingName = Coalesce(System.User.DisplayName, "there")
```

```powerfx
Global.IsPremiumUser = Topic.AccountTier = "Premium"
```

```powerfx
Topic.ApiBase = Environment.CrmApiBaseUrl
```

## Comparison Operators

| Operator | Meaning | Example |
|---|---|---|
| `=` | Equal | `Topic.Count = 0` |
| `<>` | Not equal | `Topic.Status <> "Closed"` |
| `<` | Less than | `Topic.Amount < 1000` |
| `<=` | Less than or equal | `Topic.Score <= 70` |
| `>` | Greater than | `Topic.RiskScore > 80` |
| `>=` | Greater than or equal | `Topic.DaysOpen >= 30` |
| `in` | Case-insensitive containment/membership | `"vip" in Lower(Topic.Tags)` |
| `exactin` | Case-sensitive containment/membership | `"VIP" exactin Topic.Tags` |

## Error Handling

### IsBlankOrError

```powerfx

```

### IfError with fallback

```powerfx
IfError(
```

### Error propagation strategy

Pattern:

1. Parse source value with `IfError`.
2. Store explicit error text in topic variable.
3. Branch conversation flow on whether error variable is blank.

```powerfx

```


## Delegation and Limits in Copilot Studio

Copilot Studio formula evaluation is typically in-memory for topic context data and action outputs.

Important constraints:

- Delegation warnings commonly seen in canvas apps are not surfaced the same way in Copilot Studio authoring.
- Formula length limit in many Copilot Studio expression surfaces is approximately 1000 characters.
- Break long logic into staged variables (`Topic.Step1`, `Topic.Step2`) to remain maintainable.

Guidance:

1. Keep each formula single-purpose.
2. Normalize input once and reuse.
3. Convert untyped JSON immediately after parse.
4. Avoid deeply nested `If` expressions; prefer `Switch` or `With` decomposition.

## Troubleshooting Guide

### Symptom: Formula returns blank unexpectedly

Checks:

1. Verify source variable is initialized in topic path.
2. Confirm casing for property names from parsed JSON.
3. Use `Coalesce` to guard blank values.

### Symptom: ParseJSON fields not accessible

Checks:

1. Validate raw JSON shape in test transcript.
2. Wrap arrays with `Table(...)` before `ForAll`.
3. Convert untyped fields using `Text/Value/Boolean`.

### Symptom: Formula too long

Mitigations:

- Split logic into chained `Topic.*` assignments.
- Use `With` for local readability and reuse.
- Replace repeated expressions with intermediate variables.

### Symptom: Incorrect date interpretation

Mitigations:

- Normalize to ISO date strings before `DateValue`.
- Use explicit `Text` formatting with locale token.
- Clarify timezone assumptions in user prompts.

## Appendix B: Dataverse Design Anti-Patterns

| Anti-Pattern | Why It Fails in Production | Better Alternative |
|---|---|---|
| Random naming across solutions | Breaks discoverability and API consistency | Enforce publisher prefix and schema convention policy |
| Overuse of local choices | Inconsistent semantics and reporting pain | Promote reusable vocab to global choices |
| Unreviewed cascade delete | Accidental destructive deletes | Use restrict/remove link and explicit archive process |
| Security role sprawl | Hard to audit, risky privilege drift | Role templates + periodic access reviews |
| Fat formulas with deep nesting | Hard to debug within length constraints | Compose with `With` and staged topic variables |
| Parsing JSON without casting | Type errors and blank propagation | Immediately cast with `Text`, `Value`, `Boolean` |

## Appendix E: Dataverse Deep-Dive Checklists

### Table Modeling Checklist

1. Define business capability and ownership boundary for each table.
2. Document whether table is transactional, master data, reference data, or event data.
3. Validate user/team ownership requirements before table creation.
4. Design alternate keys for integration upsert patterns.
5. Add audit requirement classification (required, optional, prohibited).
6. Validate lifecycle state model (`statecode`, `statuscode`) against process stages.
7. Plan retention and archival strategy from day one.
8. Define searchability and indexing needs for core query columns.
9. Add naming and display naming localization plan.
10. Review dependency footprint in solution layers.

### Relationship Modeling Checklist

1. Confirm directionality and cardinality with business owners.
2. Decide cascade behavior per operation with explicit risk review.
3. Document expected delete behavior and data retention obligations.
4. Ensure relationship names are understandable in API integrations.
5. Validate polymorphic lookup necessity vs explicit typed lookups.
6. Test subgrid behavior on model-driven forms.
7. Test report joins against expected analytics tooling.
8. Avoid unnecessary N:N relationships where explicit intersect table with metadata is better.
9. Document role semantics for connection-like relationships.
10. Include relationship ownership and access implications in security review.

### Security Hardening Checklist

1. Apply least privilege baseline role templates.
2. Review privilege depths for each role quarterly.
3. Move ad-hoc direct shares to team-based access where possible.
4. Implement field security profiles for sensitive attributes.
5. Audit hierarchy security implications for manager visibility.
6. Run periodic access recertification with business approvers.
7. Monitor failed access attempts for misuse detection.
8. Separate admin, maker, and end-user duties.
9. Enforce environment strategy with DLP segmentation.
10. Document emergency access break-glass process.

### Web API Integration Checklist

1. Use managed identity or secure app registration for auth flows.
2. Centralize logical name constants in integration library.
3. Always specify `$select` to reduce payload size.
4. Implement paging using nextLink until completion.
5. Handle HTTP 429 and transient 5xx with retry/backoff.
6. Use `$batch` when reducing round trips materially improves throughput.
7. Use idempotency strategy for retried create/update operations.
8. Track change tokens for incremental sync where applicable.
9. Instrument request IDs and correlation IDs for diagnostics.
10. Validate serialization formats for date/time and choice values.
